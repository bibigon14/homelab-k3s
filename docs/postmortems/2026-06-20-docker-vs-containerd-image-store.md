# Postmortem: Code fixes silently not deployed for ~14 hours due to docker build / containerd image-store mismatch

**Date:** 2026-06-20
**Author:** DS
**Status:** Mitigated
**Severity:** SEV-2 (intended fixes silently absent from production; original symptoms continued)
**Services affected:** `alertmanager-telegram-bridge` (subsequently identified as latent risk for all locally-built images)
**Platform:** homelab kubernetes platform (k3s v1.33+, single node, default containerd runtime)

---

## Summary

Between approximately 2026-06-20 04:00 UTC and 16:24 UTC, repeated attempts to
deploy a bug fix to `alertmanager-telegram-bridge` appeared to succeed -
`docker build` reported success, `kubectl rollout restart` reported success,
the new pod transitioned to `Running` - but the running pod continued to
execute the old code. The root cause: `docker build` produces images in the
Docker daemon's image store, but k3s does not use the Docker daemon. It uses
containerd directly, with its own separate image store. The two stores are
unaware of each other, and a `latest` tag in one is not visible to the other.

Because the bridge has `imagePullPolicy: Never` and uses the floating `:latest`
tag, k3s found a previously-imported image with the same name in containerd's
store and kept using it across every "redeploy". The pod restarted, the image
ID in containerd's store did not change, the symptoms in production did not
change.

The mismatch was caught only when comparing image digests directly between
`docker images` (showing a recently-built image) and
`kubectl get pod ... -o jsonpath='{.status.containerStatuses[0].imageID}'`
(showing a different, older sha256). The fix is to explicitly bridge the two
image stores via `docker save | k3s ctr images import`.

## Impact

- **Duration:** approximately 14 hours of unnecessary investigation and a
  ~12-hour window in which warning-severity alert resolutions continued to be
  silently dropped (intended fix was applied to source but not running in
  production).
- **Operator time:** an estimated 1.5 hours of debugging spent on what was
  perceived to be a deeper code-level problem before the image mismatch was
  considered.
- **Confidence:** moderate. The investigation produced several plausible-but-wrong
  theories (Telegram API caching, polling-thread bugs) that wasted time and
  could have led to bad fixes if pursued further.

## Root cause

**k3s ships with containerd as its default and only container runtime; it does
not use the Docker daemon for any image lookups.** When a pod is scheduled,
the kubelet asks containerd whether the image referenced by the Deployment
exists. Containerd consults its own image store
(`/var/lib/rancher/k3s/agent/containerd/...`). If the image is present (under
any prior provenance - initial pull, prior import, prior `crictl pull`),
containerd uses it. The Docker daemon's image store
(`/var/lib/docker/...`) is irrelevant to this lookup.

Because the Deployment specifies `imagePullPolicy: Never`, containerd never
attempts to fetch a fresher copy from a registry. It uses the local copy
unchanged. `:latest` is a tag, not a content reference; the same tag pointing
to different content in different stores is silently divergent.

## Contributing factors

1. **`imagePullPolicy: Never` is necessary for local builds, but masks staleness.**
   With `Always` or `IfNotPresent` and a real registry, the registry would
   serve as a single source of truth. `Never` accepts whatever happens to be
   local, with no warning if "local" is in the wrong store.
2. **`:latest` tag hides image identity from the operator.** A digest-pinned
   tag (`v0.3.2`, `git-abc1234`) would have surfaced the divergence at the
   `docker build` step - the operator would have noticed the rebuild produced
   a new digest while the Deployment still referenced the old tag.
3. **`kubectl rollout restart` is not a deploy.** It restarts pods of an existing
   Deployment using the existing image reference. It does not pull, rebuild,
   or otherwise refresh container images. Operator's mental model conflated
   "rollout" with "redeploy with latest code"; on Kubernetes-with-real-registries
   these coincide if the registry has been updated, but on this platform they
   diverge.
4. **`docker build` produces no warning when used with a non-Docker runtime.**
   The tool succeeds, prints a sha256, exits 0. There is no signal that the
   image will not be reachable by the host's container runtime.
5. **`docker compose` autogenerates image names that don't match operator
   expectations.** When the operator typed `docker build -t wc2026bot:latest .`,
   the Deployment was looking for `wc2026-telegram-bot-wc2026bot:latest` (the
   compose-project-name + service-name format that was originally used to
   build the image). This pattern is well-documented but easy to miss.
6. **No deploy-verification check.** A simple post-deploy assertion - `kubectl
   exec ... grep <known-string-from-new-code>` - would have caught the
   mismatch immediately after the very first attempted deploy.

## Detection

The bug was detected only by direct inspection of running container contents
(`kubectl exec ... grep ...`), which the operator only ran on the third
attempt at deploying the same fix, because the symptoms had not changed
across multiple deploys.

Earlier, more circumstantial signals existed but were not recognized:

- ImageIDs in `kubectl describe pod` were not consulted against `docker images`.
- `sudo k3s ctr images ls` was not consulted at all until the root cause was
  already suspected.

## What helped

- **`kubectl exec ... grep` of a known new identifier.** Direct, irrefutable.
  Should be the default verification step after any local-build deploy.
- **`docker save | k3s ctr images import`.** A reliable one-shot bridge
  between the two image stores. No registry required.
- **Cross-checking image digests** between `docker images` (Docker store) and
  `sudo k3s ctr images ls` (containerd store). Made the divergence explicit
  and unambiguous.

## What did not help

- **Re-reading the source code.** Multiple times. The fix was correct; the
  problem was that the fix was not in production.
- **`kubectl rollout restart`.** Restarting a pod whose image is the same image
  produces the same code. Repeating this expecting different results is the
  textbook definition of futility.
- **Increasing log verbosity and reading more logs.** The logs were correct
  for the code that was running; the code that was running was just the wrong
  code.

## Takeaways

- **The container runtime that Kubernetes uses is not necessarily Docker.**
  On k3s, kind, minikube (with containerd driver), GKE Autopilot, and most
  modern managed platforms, `docker build` and the cluster's image store are
  separate worlds. This is a fundamental detail of the platform, not a
  configuration choice.
- **Floating tags + `imagePullPolicy: Never` is a known anti-pattern for any
  multi-step deploy workflow.** It works when the build and the runtime share
  a daemon. It silently fails otherwise.
- **`kubectl rollout restart` is not a deploy.** It is a pod recycle. If the
  image reference in the Deployment spec hasn't changed, the running code
  cannot change either.
- **After any deploy of locally-built code, the very first check should be
  "is the new code in the running container?", not "are the logs
  different?".** A `grep` against `kubectl exec` is faster than reading logs,
  and gives a yes/no answer rather than a "hmm, looks similar to before".
- **Errors that look identical across deploys are a signal that the deploys
  are not changing anything.** Persisting bug-for-bug consistency after a
  "fix" is much more diagnostic than "still buggy in a slightly different
  way".
